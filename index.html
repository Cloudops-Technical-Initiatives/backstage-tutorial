Module 0

Orientation — the mental model

Backstage is not an application you install and use as-is. It is a framework: a CLI scaffolds a starter monorepo that you then own and evolve, the same way create-react-app gives you a project rather than a finished product. Think of it as a base Helm chart you fork and customise, not a SaaS you sign into.

The two packages you will live in

A scaffolded project is a Yarn monorepo. Two folders matter most:

  • packages/app — the frontend (React). Entry point is a createApp() call. You rarely touch React directly thanks to the new frontend system (below).
  • packages/backend — the backend (Node). Entry point is packages/backend/src/index.ts, which calls createBackend() and then a list of backend.add(import(...)) lines — one per plugin or module. This file is where almost all of your "light code" lives.

Configuration for both sides lives in app-config.yaml (committed defaults) and app-config.local.yaml (your machine, git-ignored). Treat these like values.yaml.

The four concepts to internalise

ConceptWhat it isWhere it's defined
Software CatalogThe inventory of everything you run — services, APIs, resources, teams.catalog-info.yaml files (entities)
Software Templates
(Scaffolder)		Forms that generate scaffolded repos/resources from a golden path.template.yaml files
TechDocsMarkdown docs that live with the code and render on the service page.mkdocs.yml + docs/
Plugins / ModulesAdd-ons. A plugin is a feature; a module augments a plugin.npm packages + app-config.yaml
Current architecture Since v1.42 the New Frontend System is the default. Plugins are integrated declaratively — you add a dependency and enable or configure it in app-config.yaml, with no TypeScript required in most cases. The New Backend System (the previous one has been removed) reduces backend wiring to a single backend.add(import(...)) line per feature. Most configuration work is YAML.
LAB-00

Predict the layout

Config
⏱ 5 min · difficulty ●○○
Objective: Before scaffolding anything, write down where each of the following belongs: (a) the React entry point, (b) the backend plugin list, (c) the file you'd edit to point the portal at a database, (d) the file describing a single microservice.
▸ Reveal solution answer
  • (a) React entry point → packages/app/src/index.ts (calls createApp from @backstage/frontend-defaults).
  • (b) Backend plugin list → packages/backend/src/index.ts (a series of backend.add(import('@backstage/plugin-…'))).
  • (c) Database / portal config → app-config.yaml (override locally in app-config.local.yaml).
  • (d) A single microservice → that service's own catalog-info.yaml, ideally in its repo next to the code.

If you guessed YAML for (c) and (d), you've already got the core instinct: describe, don't code.

Module 1

Stand it up

You'll scaffold a fresh portal and run it. SQLite is the default database, so you need nothing external for development.

Prerequisites

ToolVersion / note
Node.jsAn Active LTS release. In mid-2026 use Node 22 (safe default) or 24; Node 20 reached end-of-life in April 2026. Backstage tracks the two most recent even LTS lines — confirm on the prerequisites page. Use nvm to switch.
YarnInstalled globally (npm i -g yarn). The project pins its own Yarn version after scaffolding.
GitAny recent version.
DockerOptional but recommended — TechDocs (Module 3) uses it to render Markdown. You can skip it with a local MkDocs install instead.
LAB-01

Scaffold and run your first portal

Setup
⏱ 15 min · difficulty ●○○
Objective: Create a new Backstage app, install dependencies, and open it in your browser with both the frontend and backend running.

The scaffolder command is npx @backstage/create-app@latest. When prompted, name it my-portal and accept SQLite. Then start it. The frontend runs on port 3000, the backend on 7007.

▸ Reveal solution commands

terminal

# 1. Select a supported Node LTS (nvm recommended)
nvm install 22
nvm use 22
node -v            # → v22.x

# 2. Yarn, if you don't have it
npm install -g yarn

# 3. Scaffold — creates ./my-portal in the current folder
npx @backstage/create-app@latest
#   prompt: name  → my-portal
#   prompt: db    → SQLite (default)

# 4. Install and run both processes together
cd my-portal
yarn install
yarn dev

yarn dev launches the frontend (localhost:3000) and backend (localhost:7007) as processes [0] and [1] and opens a browser tab. If it doesn't open automatically, visit port 3000 yourself.

Good to know create-app now produces a project on the New Frontend System by default — you'll see @backstage/frontend-defaults in packages/app. That's expected and current.

my-portal/ (simplified)

my-portal/

Module 2

Software Catalog

The catalog is your inventory. Each thing you track is an entity described by a YAML file — by convention catalog-info.yaml, kept in the same repo as the thing it describes. Every entity has the same envelope:

the entity envelope

apiVersion: backstage.io/v1alpha1
kind: Component          # Component | API | Resource | System | Domain | Group | User | Location
metadata:
  name: my-service      # identity
  annotations: {}      # pointers into external systems (Grafana, GitHub, PagerDuty…)
spec:
  type: service         # kind-specific fields go here

The most common kinds: Component (a unit of software), API, Resource (infra like a DB or bucket), System/Domain (grouping), and Group/User (ownership). An entity's spec.owner must point at a Group or User that exists in the catalog.

LAB-03

Register your first Component

Config
⏱ 20 min · difficulty ●●○
Objective: Describe a real service of yours (e.g. payments-api) as a Component and get it to appear in the catalog two ways — by pointing config at a local file, and via the UI import flow.

Create examples/payments-api/catalog-info.yaml with sensible metadata (tags, a link to a dashboard, a GitHub annotation). Then register it. For local development the simplest registration is a type: file location in app-config.local.yaml.

▸ Reveal solution files + config

examples/payments-api/catalog-info.yaml

apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: payments-api
  description: Handles payment capture and refunds
  annotations:
    github.com/project-slug: my-org/payments-api
  tags: [java, payments]
  links:
    - url: https://grafana.example.com/d/payments
      title: Grafana dashboard
      icon: dashboard
spec:
  type: service
  lifecycle: production      # experimental | production | deprecated
  owner: team-payments      # must resolve to a Group/User (see LAB-04)
  system: commerce

app-config.local.yaml — register the file

catalog:
  locations:
    - type: file
      target: ../../examples/payments-api/catalog-info.yaml

Restart yarn dev; the component appears under Catalog. The owner won't resolve yet — that's expected; you'll add the Group in LAB-04.

The UI way (no config edit): in the portal go to Create → Register Existing Component (URL /catalog-import), paste the URL of a catalog-info.yaml hosted in Git, and import. This is how engineers self-register day-to-day; the file-location method is handy for local labs.

LAB-04

Add ownership: a Group, a User, a System

Config
⏱ 15 min · difficulty ●●○
Objective: Make payments-api's owner resolve by defining the team-payments Group, a User who belongs to it, and the commerce System — so the relationships render on the entity page.

Tip: a single YAML file can hold multiple entities separated by ---.

▸ Reveal solution org entities

examples/org/catalog-info.yaml

apiVersion: backstage.io/v1alpha1
kind: Group
metadata:
  name: team-payments
spec:
  type: team
  children: []
---
apiVersion: backstage.io/v1alpha1
kind: User
metadata:
  name: jdoe
spec:
  memberOf: [team-payments]
---
apiVersion: backstage.io/v1alpha1
kind: System
metadata:
  name: commerce
spec:
  owner: team-payments

app-config.local.yaml — add the new location

catalog:
  locations:
    - type: file
      target: ../../examples/payments-api/catalog-info.yaml
    - type: file
      target: ../../examples/org/catalog-info.yaml

After restart, the payments-api page shows team-payments as owner and lists it under the commerce system. A fully-qualified owner reference looks like group:default/team-payments; the short form team-payments works because everything is in the default namespace.

Module 3

TechDocs

TechDocs is docs-as-code: you write Markdown in a docs/ folder, describe it with mkdocs.yml, and Backstage renders it on the service's Docs tab. The link between a catalog entity and its docs is one annotation: backstage.io/techdocs-ref.

Three pieces sit together in the component's repo, alongside its catalog-info.yaml:

repo layout for a documented component

payments-api/
├── catalog-info.yaml
├── mkdocs.yml
└── docs/
    └── index.md
LAB-05

Attach docs to a service

Config
⏱ 25 min · difficulty ●●○
Objective: Add MkDocs to payments-api, link it with the techdocs-ref annotation, configure TechDocs for local development, and view the rendered Docs tab in the portal.

You'll need a generator. The default uses Docker; if you'd rather not run Docker, install MkDocs locally and switch the generator to local.

▸ Reveal solution docs + config

examples/payments-api/mkdocs.yml

site_name: payments-api
nav:
  - Home: index.md
  - Runbook: runbook.md
plugins:
  - techdocs-core   # optional — Backstage adds it automatically if omitted

examples/payments-api/docs/index.md

# Payments API

Captures and refunds payments. Owned by **team-payments**.

## SLOs
- p99 latency < 250ms
- availability 99.95%

examples/payments-api/catalog-info.yaml — add the annotation

metadata:
  name: payments-api
  annotations:
    backstage.io/techdocs-ref: dir:.   # docs live in this same directory

app-config.yaml — TechDocs for local dev

techdocs:
  builder: 'local'          # Backstage builds docs on demand (dev only)
  generator:
    runIn: 'docker'         # default; needs Docker running
  publisher:
    type: 'local'
No Docker? Set generator.runIn: 'local' and install MkDocs into your environment: 
# install mkdocs-techdocs-core LAST, after any other python deps
pip install mkdocs-techdocs-core

Restart, open payments-api, click the Docs tab. Backstage fetches the Markdown, runs MkDocs, and renders the site with the nav you defined.

In production Documentation is not built inside the portal. You set techdocs.builder: 'external', generate the static site in CI/CD, and publish the output to object storage — S3, GCS, or Azure Blob via the publisher config. The portal then serves the prebuilt files.

Module 4

Software Templates (Scaffolder)

A template is a YAML file (kind: Template) with three jobs: a set of parameters that render as a form, a sequence of steps that each run a built-in action, and optional output links. It's a golden path: fill the form, get a scaffolded, registered, ready-to-go service. The format version is scaffolder.backstage.io/v1beta3.

Templates show up under Create (/create). The full list of actions available in your instance is browsable at /create/actions — common ones are fetch:template (render a skeleton), publish:github / publish:gitlab (create the repo), and catalog:register (add the new component to the catalog).

LAB-06

Build and run a golden-path template

Config
⏱ 35 min · difficulty ●●●
Objective: Create a template that asks for a service name and owner, renders a skeleton (a templated catalog-info.yaml + README), and logs the result. Register the template, open /create, and run it.

To keep this fully offline we'll use fetch:template + debug:log (no GitHub needed). The real-world publish/register steps are shown in the callout afterward.

▸ Reveal solution template + skeleton

examples/templates/new-service/template.yaml

apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: new-service
  title: New service skeleton
  description: Scaffolds a service folder with a catalog-info.yaml
spec:
  owner: team-payments
  type: service
  parameters:
    - title: Service details
      required: [name, owner]
      properties:
        name:
          title: Name
          type: string
          ui:autofocus: true
        owner:
          title: Owner
          type: string
          ui:field: OwnerPicker        # dropdown of catalog Groups
          ui:options:
            catalogFilter:
              kind: Group
  steps:
    - id: fetch
      name: Render skeleton
      action: fetch:template
      input:
        url: ./skeleton
        values:
          name: ${{ parameters.name }}
          owner: ${{ parameters.owner }}
    - id: log
      name: Report
      action: debug:log
      input:
        message: Scaffolded ${{ parameters.name }} for ${{ parameters.owner }}

examples/templates/new-service/skeleton/catalog-info.yaml

apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: ${{ values.name }}
spec:
  type: service
  lifecycle: experimental
  owner: ${{ values.owner }}

app-config.local.yaml — register the template (note the rules)

catalog:
  locations:
    - type: file
      target: ../../examples/templates/new-service/template.yaml
      rules:
        - allow: [Template]   # file locations must explicitly allow the Template kind

Restart, open Create (/create), launch New service skeleton, fill the form, and run. Watch the task log render the skeleton and print your message. Two reusable ideas to notice: ${{ parameters.x }} reads form input inside the template; ${{ values.x }} reads it inside skeleton files.

Troubleshooting On Node 20+ the scaffolder needs a flag to run templates safely. If a run fails immediately, start the dev server with: 
export NODE_OPTIONS=--no-node-snapshot
yarn dev
The real golden path In a live portal you'd swap debug:log for the actions that make it real — exactly the automation you already write by hand: 
- id: publish
  action: publish:github
  input:
    repoUrl: github.com?owner=my-org&repo=${{ parameters.name }}
- id: register
  action: catalog:register
  input:
    repoContentsUrl: ${{ steps.publish.output.repoContentsUrl }}
    catalogInfoPath: /catalog-info.yaml
A template that wraps publish:github and catalog:register turns repository creation, boilerplate, and catalog onboarding into a single self-service form — no manual setup required.

Module 5

Install a plugin — config only

This is where the New Frontend System pays off. Installing a plugin is two moves: add the package as a dependency, then (optionally) configure it in app-config.yaml. Feature discovery wires it in automatically — no editing React code.

New apps ship with discovery on:

app-config.yaml — feature discovery (default in new apps)

app:
  packages: all     # auto-install any plugin added to packages/app deps

Most community plugins now live under the @backstage-community/ npm scope. We'll add the GitHub Actions plugin so a service page shows its CI/CD workflow runs.

LAB-07

Add a CI/CD view to a service

Config
⏱ 25 min · difficulty ●●○
Objective: Install the GitHub Actions frontend plugin, configure a GitHub integration so the portal can read workflow runs, and annotate payments-api so its page surfaces CI/CD.
▸ Reveal solution install + config

terminal — add the dependency to the frontend package

yarn --cwd packages/app add @backstage-community/plugin-github-actions

With app.packages: all, discovery picks it up on the next start — no code change in App.tsx.

app-config.yaml — let the backend talk to GitHub

integrations:
  github:
    - host: github.com
      token: ${GITHUB_TOKEN}   # a PAT, supplied via env var — never hard-code

payments-api/catalog-info.yaml — point at the repo

metadata:
  annotations:
    github.com/project-slug: my-org/payments-api

Export the token and restart:

export GITHUB_TOKEN=ghp_xxx
yarn dev

The component page now exposes a CI/CD view backed by GitHub Actions for that repo.

Enable / disable / tune — still just YAML Every visual piece is an "extension" you can control under app.extensions. The schema: 
app:
  extensions:
    - <extension-id>:          # full form
        disabled: <true|false>
        config: <plugin-specific>
    - <extension-id>: false     # shorthand: turn one off
Extension IDs depend on what's installed — discover them from the plugin's README or by inspecting the running app rather than copying IDs blind. The point: showing, hiding, and configuring features is configuration, not code.
LAB-07b

Wire up the Kubernetes plugin

StretchConfig
⏱ 30 min · difficulty ●●●
Objective: Surface live workloads on a service page by configuring the Kubernetes plugin against a cluster you control.

Kubernetes is a frontend + backend plugin. Install both halves, declare your cluster(s) in config, and annotate the component with a selector. This mirrors how you already register clusters with tooling.

▸ Reveal solution k8s wiring

terminal

yarn --cwd packages/app add @backstage/plugin-kubernetes
yarn --cwd packages/backend add @backstage/plugin-kubernetes-backend

The frontend half is auto-discovered. The backend half is one line in the backend index (you'll get comfortable with this pattern in Module 6):

packages/backend/src/index.ts

backend.add(import('@backstage/plugin-kubernetes-backend'));

app-config.yaml — declare clusters

kubernetes:
  serviceLocatorMethod:
    type: multiTenant
  clusterLocatorMethods:
    - type: config
      clusters:
        - name: prod
          url: ${K8S_API_URL}
          authProvider: serviceAccount
          serviceAccountToken: ${K8S_SA_TOKEN}

component annotation — which workloads belong to this service

metadata:
  annotations:
    backstage.io/kubernetes-id: payments-api

Label your Deployments/Pods with backstage.io/kubernetes-id: payments-api and the service page shows their live status. Everything portal-side is config — the only "code" was one backend import line.

Module 6

Light backend code

Eventually "configure it" runs out and you need to add capability. In the New Backend System that means working in one file — packages/backend/src/index.ts — which is a list of features:

packages/backend/src/index.ts (shape)

import { createBackend } from '@backstage/backend-defaults';

const backend = createBackend();
backend.add(import('@backstage/plugin-app-backend'));
backend.add(import('@backstage/plugin-catalog-backend'));
backend.add(import('@backstage/plugin-scaffolder-backend'));
// add more plugins/modules here, one per line
backend.start();

Two nouns: a plugin is a standalone feature (catalog, scaffolder); a module augments a plugin (e.g. extra scaffolder actions, a catalog processor). You install either the same way — a dependency plus one backend.add(import(...)) line.

terminal

LAB-09

Write a tiny custom Scaffolder action

CodeStretch
⏱ 45 min · difficulty ●●●
Objective: Create your own action, acme:echo, that takes a message and logs it, then call it from a template step. This is the smallest meaningful piece of "real" Backstage code — and the pattern scales to actions that hit your internal APIs.

Scaffold a module package with the CLI, then add an action and register it on the scaffolder's extension point. The CLI does the boilerplate:

terminal — generate the module package

yarn backstage-cli new
# choose:  scaffolder-module   →   name: acme
▸ Reveal solution action + module

The CLI creates plugins/scaffolder-backend-module-acme/. Define the action:

plugins/scaffolder-backend-module-acme/src/actions/echo.ts

import { createTemplateAction } from '@backstage/plugin-scaffolder-node';

export const createEchoAction = () =>
  createTemplateAction<{ message: string }>({
    id: 'acme:echo',
    schema: {
      input: {
        type: 'object',
        required: ['message'],
        properties: { message: { type: 'string' } },
      },
    },
    async handler(ctx) {
      ctx.logger.info(`Echo: ${ctx.input.message}`);
      ctx.output('echoed', ctx.input.message);
    },
  });

plugins/scaffolder-backend-module-acme/src/module.ts — register it

import { createBackendModule } from '@backstage/backend-plugin-api';
import { scaffolderActionsExtensionPoint }
  from '@backstage/plugin-scaffolder-node/alpha';
import { createEchoAction } from './actions/echo';

export const scaffolderModuleAcme = createBackendModule({
  pluginId: 'scaffolder',      # which plugin we augment
  moduleId: 'acme',
  register(reg) {
    reg.registerInit({
      deps: { scaffolder: scaffolderActionsExtensionPoint },
      async init({ scaffolder }) {
        scaffolder.addActions(createEchoAction());
      },
    });
  },
});

packages/backend/src/index.ts — install your module

backend.add(import('@internal/plugin-scaffolder-backend-module-acme'));

use it in a template step

- id: say
  action: acme:echo
  input:
    message: 'hello from my first action'

Restart, run a template using the step, and watch your message in the task log. The same handler could call an internal REST API, open a Jira ticket, or trigger a pipeline — that's how teams extend the golden path to their own systems.

Version note Import paths for extension points (e.g. /alpha sub-paths) and the exact action-schema helpers shift between releases. If an import doesn't resolve, check the installed @backstage/plugin-scaffolder-node version and the current "Writing Custom Actions" doc rather than assuming this snippet is eternal. Also remember to use camelCase for action IDs in id references inside templates — dashes get read as minus signs in ${{ … }} expressions.

Module 7

Capstone

No walkthrough this time — just an acceptance checklist. Combine everything into one coherent slice of a portal. Use a service that's real to your team.

Deliverable: A running portal where one service is fully wired end-to-end, plus a template that produces more services like it.

Acceptance criteria

  • A Group and at least one User exist, and a Component resolves its owner to that Group (Modules 2).
  • The Component has working TechDocs on its Docs tab (Module 3).
  • A Template registered under /create scaffolds a new Component of the same shape, with an OwnerPicker form field (Module 4).
  • At least one plugin (GitHub Actions or Kubernetes) shows live data on the Component page, configured entirely in app-config.yaml (Module 5).
  • At least one backend module is registered via backend.add(import(...)) — the GitHub scaffolder module counts (Module 6).
Stretch goal Add your acme:echo action from LAB-09 as a final step in the capstone template, logging which team requested the new service. You'll have touched catalog, docs, templates, plugins, config, and code — the full surface of day-to-day Backstage work.

Appendix

Reference & survival kit

Keeping current

Backstage ships a new release on the Tuesday before the third Wednesday of every month. Stay close to the latest line and upgrade with the CLI rather than editing versions by hand:

# bumps all @backstage packages to the latest main release line, together
yarn backstage-cli versions:bump

Read the release notes before a bump — that's where breaking changes and migration steps are documented.

Troubleshooting cheatsheet

SymptomLikely fix
Template run fails instantlyexport NODE_OPTIONS=--no-node-snapshot before yarn dev (Node 20+).
Docs tab spins / errorsDocker not running, or no local MkDocs. Set generator.runIn to match what you have; pip install mkdocs-techdocs-core for local.
Owner shows as unresolvedThe Group/User in spec.owner isn't in the catalog yet — register it.
Installed a plugin, nothing appearsConfirm app.packages: all (feature discovery), and that you added the dep to packages/app (frontend) or packages/backend (backend) as appropriate.
Template not in /createFile location needs rules: [{ allow: [Template] }].
Backend won't read GitHub/cloud credsValues come from env vars referenced as ${VAR} in app-config; make sure the var is exported in the shell running the backend.
