Module 0 · Orientation

Orientation — the mental model

Backstage is not an application you install and use as-is. It is a framework: a CLI scaffolds a starter monorepo that you then own and evolve, the same way create-react-app gives you a project rather than a finished product. Think of it as a base Helm chart you fork and customise, not a SaaS you sign into.

The two packages you will live in

A scaffolded project is a Yarn monorepo. Two folders matter most:

  • packages/app — the frontend (React). Entry point is a createApp() call. You rarely touch React directly thanks to the new frontend system (below).
  • packages/backend — the backend (Node). Entry point is packages/backend/src/index.ts, which calls createBackend() and then a list of backend.add(import(...)) lines — one per plugin or module. This file is where almost all of your "light code" lives.

Configuration for both sides lives in app-config.yaml (committed defaults) and app-config.local.yaml (your machine, git-ignored). Treat these like values.yaml.

The four concepts to internalise

ConceptWhat it isWhere it's defined
Software CatalogThe inventory of everything you run — services, APIs, resources, teams.catalog-info.yaml files (entities)
Software Templates
(Scaffolder)		Forms that generate scaffolded repos/resources from a golden path.template.yaml files
TechDocsMarkdown docs that live with the code and render on the service page.mkdocs.yml + docs/
Plugins / ModulesAdd-ons. A plugin is a feature; a module augments a plugin.npm packages + app-config.yaml
Current architecture Since v1.42 the New Frontend System is the default. Plugins are integrated declaratively — you add a dependency and enable or configure it in app-config.yaml, with no TypeScript required in most cases. The New Backend System (the previous one has been removed) reduces backend wiring to a single backend.add(import(...)) line per feature. Most configuration work is YAML.

Stand it up

You'll scaffold a fresh portal and run it. SQLite is the default database, so you need nothing external for development.

Set up a GitHub Personal Access Token

Several labs need the portal to read from (and later write to) your GitHub. Create a token once and keep it in an environment variable — never paste it into a file.

  1. GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens → Generate new token.
  2. Give it access to the repo you'll create in Module 2 (or "All repositories" for the labs).
  3. Repository permissions: Contents = Read and write, Issues = Read and write, Pull requests = Read and write, Administration = Read and write (needed only if a template will create new repos).
  4. Generate, copy it, and export it in the terminal where you run the portal:
# keep this in your shell profile or a local .env you don't commit
export GITHUB_TOKEN=github_pat_xxxxxxxx

Install Docker Desktop (macOS)

TechDocs renders Markdown inside a container, and you'll run Jenkins locally in Docker. Install Docker Desktop for Mac:

# option A — Homebrew
brew install --cask docker

# option B — download the .dmg for Apple Silicon or Intel from docker.com
#   https://www.docker.com/products/docker-desktop/

# then LAUNCH Docker Desktop once (menu-bar whale must be running), and verify:
docker run --rm hello-world
Note Docker Desktop must be open and running (not just installed) for the TechDocs and Jenkins labs — the whale icon should be in your macOS menu bar.

Module 2 of 7

Software Catalog

The catalog is your inventory. Each thing you track is an entity described by a YAML file — by convention catalog-info.yaml, kept in the same repo as the thing it describes. Every entity has the same envelope:

the entity envelope

apiVersion: backstage.io/v1alpha1
kind: Component          # Component | API | Resource | System | Domain | Group | User | Location
metadata:
  name: my-service      # identity
  annotations: {}      # pointers into external systems (GitHub, Jenkins, dashboards…)
spec:
  type: service         # kind-specific fields go here

The most common kinds: Component (a unit of software), API, Resource (infra like a DB or bucket), System/Domain (grouping), and Group/User (ownership). An entity's spec.owner must point at a Group or User that exists in the catalog.

LAB-03

Register a Component from your GitHub repo

Config
25 min · Medium
Your task: Create a real GitHub repo with a catalog-info.yaml, connect the portal to GitHub with your PAT, and register the component from the UI by pasting the file's URL.

This uses only GitHub — no cloud, no dummy links. Throughout the labs, replace your-org/demo-service with the repo you actually create.

▸ Reveal solution repo + config + register

1 · create a GitHub repo and add this file at its root

On GitHub, create a repo (e.g. demo-service). Add a file named catalog-info.yaml:

apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: demo-service
  description: My first catalog entry
  annotations:
    github.com/project-slug: your-org/demo-service   # your real org/repo
  tags: [demo, platform]
spec:
  type: service
  lifecycle: experimental   # experimental | production | deprecated
  owner: platform-team       # resolved in LAB-04

2 · app-config.local.yaml — connect the portal to GitHub

integrations:
  github:
    - host: github.com
      token: ${GITHUB_TOKEN}   # the PAT you exported in Module 1

Restart with the token exported (export GITHUB_TOKEN=… && yarn dev).

3 · register it from the UI

In the portal go to Create → Register Existing Component (/catalog-import) and paste the full URL to the file, for example https://github.com/your-org/demo-service/blob/main/catalog-info.yaml. Analyze → Import. The component now appears under Catalog. The owner won't resolve yet — you'll fix that in LAB-04.

Local-file alternative If you'd rather not push to GitHub yet, drop the same file at examples/demo-service/catalog-info.yaml and point config at it instead: 
catalog:
  locations:
    - type: file
      target: ../../examples/demo-service/catalog-info.yaml
The GitHub route is closer to how engineers self-register day-to-day, so prefer it once your token works.

Module 3 of 7

TechDocs

TechDocs is docs-as-code: you write Markdown in a docs/ folder, describe it with mkdocs.yml, and Backstage renders it on the service's Docs tab. The link between a catalog entity and its docs is one annotation: backstage.io/techdocs-ref.

Three pieces sit together in the component's repo, alongside its catalog-info.yaml:

repo layout for a documented component

demo-service/
├── catalog-info.yaml
├── mkdocs.yml
└── docs/
    └── index.md
LAB-05

Attach docs to a service

Config
25 min · Medium
Your task: Add MkDocs to demo-service, link it with the techdocs-ref annotation, point TechDocs at Docker for rendering, and view the rendered Docs tab.

This uses the Docker Desktop you installed in Module 1 — make sure it's running. A local MkDocs install is shown as a fallback.

▸ Reveal solution docs + config

Add these two files next to the repo's catalog-info.yaml (in the GitHub repo, or in examples/demo-service/ if you used the local-file route).

mkdocs.yml

site_name: demo-service
nav:
  - Home: index.md
  - Runbook: runbook.md
plugins:
  - techdocs-core   # optional — Backstage adds it automatically if omitted

docs/index.md

# demo-service

My first service in the developer portal, owned by **platform-team**.

## Notes
- This page is rendered from Markdown by TechDocs.
- Edit the Markdown, refresh, and the Docs tab updates.

catalog-info.yaml — add the annotation

metadata:
  name: demo-service
  annotations:
    backstage.io/techdocs-ref: dir:.   # docs live in this same directory

app-config.yaml — TechDocs for local dev (Docker)

techdocs:
  builder: 'local'          # Backstage builds docs on demand (dev only)
  generator:
    runIn: 'docker'         # uses your running Docker Desktop
  publisher:
    type: 'local'
Prefer no Docker? Set generator.runIn: 'local' and install MkDocs instead: 
# install mkdocs-techdocs-core LAST, after any other python deps
pip install mkdocs-techdocs-core

Restart, open demo-service, click the Docs tab. Backstage fetches the Markdown, runs MkDocs (in Docker), and renders the site with your nav.

In production Documentation is not built inside the portal. You set techdocs.builder: 'external', generate the static site in CI/CD, and publish the output to object storage via the publisher config. The portal then serves the prebuilt files.
✓You've built docs-as-code for your service, rendered on its Docs tab through Docker.
Module 4 of 7

Software Templates (Scaffolder)

A template is a YAML file (kind: Template) with three jobs: a set of parameters that render as a form, a sequence of steps that each run a built-in action, and optional output links. It's a golden path: fill the form, get a scaffolded, registered, ready-to-go service. The format version is scaffolder.backstage.io/v1beta3.

Templates show up under Create (/create). The full list of actions available in your instance is browsable at /create/actions — common ones are fetch:template (render a skeleton), publish:github (create the repo), and catalog:register (add the new entity to the catalog).

We'll build a practical one: a self-service "Request infrastructure" form. It collects a few choices, generates a Terraform stub plus a catalog entry, and opens a new GitHub repo for review. No cloud access is needed — the template produces the IaC and a repo; the actual terraform apply would run later in a pipeline.

LAB-06

Build a self-service "Request infrastructure" template

Config
40 min · Hard
Your task: Create a template whose form collects a name, owner, cloud provider, environment, and size; generates a Terraform stub + a Resource catalog entry + README; then publishes a new GitHub repo and registers it in the catalog.

This needs your GitHub PAT (Module 1). publish:github is provided by the GitHub scaffolder module — the starter usually includes it; if /create/actions doesn't list it, add it via Module 6 (the one-line pattern in LAB-08).

▸ Reveal solution template + skeleton

examples/templates/request-infra/template.yaml

apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: request-infra
  title: Request infrastructure
  description: Generate a Terraform stub and open a repo for review
spec:
  owner: platform-team
  type: resource
  parameters:
    - title: Request details
      required: [name, owner, cloud, environment, size]
      properties:
        name:
          title: Resource name
          type: string
          ui:autofocus: true
        owner:
          title: Owner
          type: string
          ui:field: OwnerPicker
          ui:options:
            catalogFilter:
              kind: Group
        cloud:
          title: Cloud provider
          type: string
          enum: [aws, gcp, azure]
        environment:
          title: Environment
          type: string
          enum: [dev, staging, prod]
        size:
          title: Size
          type: string
          enum: [small, medium, large]
  steps:
    - id: fetch
      name: Render Terraform stub
      action: fetch:template
      input:
        url: ./skeleton
        values:
          name: ${{ parameters.name }}
          owner: ${{ parameters.owner }}
          cloud: ${{ parameters.cloud }}
          environment: ${{ parameters.environment }}
          size: ${{ parameters.size }}
    - id: publish
      name: Open a GitHub repo
      action: publish:github
      input:
        repoUrl: github.com?owner=your-org&repo=${{ parameters.name }}
        description: Infra request for ${{ parameters.name }}
    - id: register
      name: Register in catalog
      action: catalog:register
      input:
        repoContentsUrl: ${{ steps.publish.output.repoContentsUrl }}
        catalogInfoPath: /catalog-info.yaml
  output:
    links:
      - title: Open the repo
        url: ${{ steps.publish.output.remoteUrl }}
      - title: Open in catalog
        icon: catalog
        entityRef: ${{ steps.register.output.entityRef }}

examples/templates/request-infra/skeleton/main.tf

# Terraform stub generated by the developer portal.
# apply happens in CI/CD, not from the portal.
terraform {
  required_version = ">= 1.6"
}

# cloud:       ${{ values.cloud }}
# environment: ${{ values.environment }}
# size:        ${{ values.size }}

# TODO: call your real module, e.g.
# module "${{ values.name }}" {
#   source = "git::https://github.com/your-org/terraform-modules//compute"
# }

examples/templates/request-infra/skeleton/catalog-info.yaml

apiVersion: backstage.io/v1alpha1
kind: Resource
metadata:
  name: ${{ values.name }}
spec:
  type: cloud-resource
  owner: ${{ values.owner }}
  system: internal-tools

app-config.local.yaml — register the template (note the rules)

catalog:
  locations:
    - type: file
      target: ../../examples/templates/request-infra/template.yaml
      rules:
        - allow: [Template]   # file locations must explicitly allow the Template kind

Restart, open Create (/create), launch Request infrastructure, fill the form, and run. The task creates the repo with your Terraform stub and registers the new Resource. Two reusable ideas: ${{ parameters.x }} reads form input in the template; ${{ values.x }} reads it inside skeleton files.

Troubleshooting If a run fails instantly on Node 22/24, start the dev server with the scaffolder flag: 
export NODE_OPTIONS=--no-node-snapshot
yarn dev
And if /create/actions doesn't list publish:github, register the GitHub scaffolder module (Module 6, LAB-08 shows the one-line pattern).
More GitOps-style variant Instead of creating a brand-new repo, point a single central infra repo and open a Pull Request into it (using the GitHub PR scaffolder actions) so changes are reviewed before merge — then a pipeline runs terraform plan/apply on merge. Same template, different publish step.

Module 5 of 7

Install a plugin — config only

This is where the New Frontend System pays off. Installing a plugin is two moves: add the package as a dependency, then (optionally) configure it in app-config.yaml. Feature discovery wires it in automatically — no editing React code.

New apps ship with discovery on:

app-config.yaml — feature discovery (default in new apps)

app:
  packages: all     # auto-install any plugin added to packages/app deps

Most community plugins now live under the @backstage-community/ npm scope. We'll add the GitHub Actions plugin so a service page shows its CI/CD workflow runs.

Enable / disable / tune — still just YAML Every visual piece is an "extension" you can control under app.extensions. The schema: 
app:
  extensions:
    - <extension-id>:          # full form
        disabled: <true|false>
        config: <plugin-specific>
    - <extension-id>: false     # shorthand: turn one off
Extension IDs depend on what's installed — discover them from the plugin's README or by inspecting the running app rather than copying IDs blind. Showing, hiding, and configuring features is configuration, not code.

Module 6 of 7

Light backend code

Eventually "configure it" runs out and you need to add capability. In the New Backend System that means working in one file — packages/backend/src/index.ts — which is a list of features:

packages/backend/src/index.ts (shape)

import { createBackend } from '@backstage/backend-defaults';

const backend = createBackend();
backend.add(import('@backstage/plugin-app-backend'));
backend.add(import('@backstage/plugin-catalog-backend'));
backend.add(import('@backstage/plugin-scaffolder-backend'));
// add more plugins/modules here, one per line
backend.start();

Two nouns: a plugin is a standalone feature (catalog, scaffolder); a module augments a plugin (e.g. extra scaffolder actions, a catalog processor). You install either the same way — a dependency plus one backend.add(import(...)) line.

LAB-09

Write a tiny custom Scaffolder action

CodeStretch
45 min · Hard
Your task: Create your own action, acme:recordRequest, that records an infrastructure request (writes a request file and logs it), then call it from the Module 4 template. This is the smallest meaningful piece of "real" Backstage code — and the same pattern hits external systems later.

Scaffold a module package with the CLI, then add an action and register it on the scaffolder's extension point. The CLI does the boilerplate:

terminal — generate the module package

yarn backstage-cli new
# choose:  scaffolder-module   →   name: acme
▸ Reveal solution action + module

The CLI creates plugins/scaffolder-backend-module-acme/. Define the action:

plugins/scaffolder-backend-module-acme/src/actions/recordRequest.ts

import { createTemplateAction } from '@backstage/plugin-scaffolder-node';

export const createRecordRequestAction = () =>
  createTemplateAction<{ name: string; cloud: string; environment: string }>({
    id: 'acme:recordRequest',
    schema: {
      input: {
        type: 'object',
        required: ['name', 'cloud', 'environment'],
        properties: {
          name: { type: 'string' },
          cloud: { type: 'string' },
          environment: { type: 'string' },
        },
      },
    },
    async handler(ctx) {
      const record = {
        request: ctx.input.name,
        cloud: ctx.input.cloud,
        environment: ctx.input.environment,
        requestedAt: new Date().toISOString(),
      };
      ctx.logger.info(`Recording infra request: ${JSON.stringify(record)}`);
      // write the record into the scaffolder workspace using Node fs
      const fs = await import('fs/promises');
      await fs.writeFile(
        `${ctx.workspacePath}/request.json`,
        JSON.stringify(record, null, 2),
      );
      ctx.output('recorded', record.request);
    },
  });

plugins/scaffolder-backend-module-acme/src/module.ts — register it

import { createBackendModule } from '@backstage/backend-plugin-api';
import { scaffolderActionsExtensionPoint }
  from '@backstage/plugin-scaffolder-node/alpha';
import { createRecordRequestAction } from './actions/recordRequest';

export const scaffolderModuleAcme = createBackendModule({
  pluginId: 'scaffolder',      # which plugin we augment
  moduleId: 'acme',
  register(reg) {
    reg.registerInit({
      deps: { scaffolder: scaffolderActionsExtensionPoint },
      async init({ scaffolder }) {
        scaffolder.addActions(createRecordRequestAction());
      },
    });
  },
});

packages/backend/src/index.ts — install your module

backend.add(import('@internal/plugin-scaffolder-backend-module-acme'));

use it as a step in the Module 4 template

- id: record
  action: acme:recordRequest
  input:
    name: ${{ parameters.name }}
    cloud: ${{ parameters.cloud }}
    environment: ${{ parameters.environment }}

Restart, run the template, and watch the request get recorded in the task log.

Make it real — open a GitHub Issue Instead of just writing a file, the handler can open a tracking ticket using the GitHub integration you already configured. Inside handler: 
// uses the same GITHUB_TOKEN integration
const res = await fetch(
  'https://api.github.com/repos/your-org/infra-requests/issues', {
    method: 'POST',
    headers: {
      Authorization: `token ${process.env.GITHUB_TOKEN}`,
      Accept: 'application/vnd.github+json',
    },
    body: JSON.stringify({
      title: `Infra request: ${ctx.input.name}`,
      body: `cloud: ${ctx.input.cloud}, env: ${ctx.input.environment}`,
    }),
  });
ctx.logger.info(`Opened request issue (status ${res.status})`);
Same shape works for any internal API — a ticketing system, an approval workflow, or a pipeline trigger.
Version note Import paths for extension points (e.g. /alpha sub-paths) and helper names on ctx shift between releases. If an import or method doesn't resolve, check the installed @backstage/plugin-scaffolder-node version and the current "Writing Custom Actions" doc. Use camelCase action IDs (acme:recordRequest) — dashes get read as minus signs in ${{ … }} expressions.

Reference & survival kit

Keeping current

Backstage ships a new release on the Tuesday before the third Wednesday of every month. Stay close to the latest line and upgrade with the CLI rather than editing versions by hand:

# bumps all @backstage packages to the latest main release line, together
yarn backstage-cli versions:bump

Read the release notes before a bump — that's where breaking changes and migration steps are documented.

Troubleshooting cheatsheet

SymptomLikely fix
Template run fails instantlyexport NODE_OPTIONS=--no-node-snapshot before yarn dev (Node 20+).
Docs tab spins / errorsDocker not running, or no local MkDocs. Set generator.runIn to match what you have; pip install mkdocs-techdocs-core for local.
Owner shows as unresolvedThe Group/User in spec.owner isn't in the catalog yet — register it.
Installed a plugin, nothing appearsConfirm app.packages: all (feature discovery), and that you added the dep to packages/app (frontend) or packages/backend (backend) as appropriate.
Template not in /createFile location needs rules: [{ allow: [Template] }].
Backend won't read GitHub/Jenkins credsValues come from env vars referenced as ${VAR} in app-config; make sure the var (e.g. GITHUB_TOKEN, JENKINS_TOKEN) is exported in the shell running the backend.
